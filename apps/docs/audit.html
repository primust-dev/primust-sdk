Audit Guide

For auditors, compliance officers, and GRC teams evaluating Primust Evidence Packs.

The Fundamental Claim

When an organization presents a Primust Evidence Pack, they are making a specific, bounded claim:

The Claim

"For every declared operation in this pipeline, we can prove — via cryptographic signature, zero-knowledge proof, or attested record — that the code which was supposed to run did run, on the data it was supposed to process, producing the output that was recorded."

Critical Disclosure

Primust proves what was declared should run, not that everything was declared. An organization could omit pipeline stages from their VPEC definition. Primust's Gap Taxonomy identifies and labels known omission categories, but the declaration boundary is set by the organization. Auditors must independently assess declaration completeness.

This distinction is fundamental. Primust can guarantee execution integrity within the declared surface — it cannot guarantee the declared surface is exhaustive. The Provable Surface Report and Gap Taxonomy exist specifically to make this boundary visible to auditors.

Five-Minute Verification

Every claim in an Evidence Pack can be independently verified using the open-source primust-verify tool. No Primust account is required.

Install

pip install primust-verify

Verify an Evidence Pack

primust verify evidence-pack-2026-03-12.json

What the Five Verification Checks Prove

Check What It Proves Failure Means
Signature The Evidence Pack was signed by the declared organization key and has not been modified since signing Pack has been tampered with or key is not recognized
Chain Every execution record links to the previous one via cryptographic hash — no records have been inserted, removed, or reordered Records have been altered after the fact
ZK Proofs For operations at Proof Level 3+, the declared computation was actually performed on the declared inputs, without revealing the inputs themselves The claimed computation did not occur as described
Timestamps All records fall within the declared execution window and are monotonically ordered Timeline has been manipulated or records are out of sequence
Gaps The organization has disclosed known coverage gaps per the Gap Taxonomy; gap count and severity are reported Undisclosed gaps or critical gaps present

Proof Levels — What Each Means for Audit

Level Name Mechanism Audit Guidance
P1 Attestation Organization asserts a fact; Primust records the assertion with a timestamp and signature Treat as a signed declaration. No independent proof of execution. Suitable for low-risk metadata. Auditors should flag if P1 is used for high-risk operations.
P2 Hash Lock Input/output hashes are recorded. If you have the original data, you can confirm it matches. Proves data was not altered after recording. Does not prove the computation itself. Request original data to verify hashes.
P3 ZK Proven A zero-knowledge proof confirms the declared computation ran on the declared inputs, producing the declared output — without revealing the data. Strongest practical proof. The verifier confirms execution integrity without needing to see underlying data. Suitable for regulated and high-risk operations.
P4 ZK + Timestamp Anchor ZK proof plus an external timestamp anchor (e.g., blockchain or RFC 3161 TSA). Adds non-repudiation on timing. Useful when "when it happened" matters as much as "what happened." Verify the external anchor independently.
P5 Full Reproducible ZK proof, timestamp anchor, and a reproducibility manifest that allows a third party to re-execute the pipeline and compare results. Maximum assurance. Auditor can independently re-run the pipeline. Requires access to the reproducibility environment. Rare in production; typically used for critical regulatory submissions.

Reading a Provable Surface Report

The Provable Surface Report shows exactly what percentage of a pipeline is covered at each proof level, and where the gaps are.

# Surface coverage is shown inside each Evidence Pack and VPEC.
# Use the dashboard at app.primust.com or inspect the VPEC JSON directly.
primust-verify vpec_loan-underwriting-v3.json --json | jq .coverage

Verifying an Evidence Pack

# Verify a VPEC artifact
primust-verify vpec_2026-03-12.json

# Verify and output machine-readable JSON
primust-verify vpec_2026-03-12.json --json

# Verify against a specific org public key (BYOK)
primust-verify vpec_2026-03-12.json --trust-root org-key-abc123.pem

# Offline verification (skip Rekor transparency log check)
primust-verify vpec_2026-03-12.json --skip-network

# Reject test-mode credentials in production audit
primust-verify vpec_2026-03-12.json --production

Framework Alignment

Primust Evidence Packs can serve as supporting evidence for multiple compliance frameworks. Below is guidance on how Primust artifacts map to specific requirements.

EU AI Act

Articles 9, 12, and 17 require risk management documentation, logging of automatic decisions, and quality management systems. Primust VPECs provide cryptographically verifiable execution logs for AI pipeline stages. The Provable Surface Report maps directly to the "technical documentation" requirement under Article 11. Note: Primust does not assess risk classification — that remains the provider's responsibility.

HIPAA

The Security Rule (§164.312) requires audit controls and integrity controls. Evidence Packs provide tamper-evident execution records. Hash-locked and ZK-proven stages satisfy integrity verification requirements. Note: Primust never transits PHI. The ZK proofs confirm computation integrity without exposing underlying health data.

SOC 2

Trust Services Criteria CC7.1 (system monitoring) and CC8.1 (change management) align with VPEC execution records and policy bundle versioning. Evidence Packs can serve as artifacts for SOC 2 Type II evidence requests. Auditors can verify artifacts independently using primust-verify.

AIUC-1

Primust implements the AIUC-1 (AI Use Case Compliance) schema natively. Pipeline init records, check records, and compliance requirement fields follow the AIUC-1 specification. The check execution record schema supports actor_id, explanation_commitment, and bias_audit fields for EU AI Act compliance. See the Schema Reference for full field details.

Critical Limitation

Primust is an evidence infrastructure, not a compliance assessment tool. It proves execution integrity for declared operations. It does not determine whether your operations are compliant with any specific framework. Compliance determination remains the responsibility of qualified assessors evaluating Primust artifacts alongside other evidence.

Red Flags

When reviewing Evidence Packs, the following conditions warrant further investigation:

Condition What It Indicates
All stages at P1 basis No cryptographic proof of execution — only signed attestations. Acceptable for low-risk workflows only. For regulated pipelines, request justification for why ZK proofs are not used.
All coverage is attestation The organization may be using Primust as a logging tool rather than a proof system. The value proposition of provable execution is absent.
Unresolved critical gaps The Gap Taxonomy classifies some gaps as critical (e.g., G-01 undeclared stage, G-05 missing ZK circuit). Critical gaps should have waivers or remediation plans.
Expired waivers Gap waivers have expiration dates. An expired waiver means a known gap has exceeded its accepted remediation timeline. Treat as an unresolved gap.
Sandbox VPECs in production evidence VPECs generated in sandbox mode are not suitable for production audit. The VPEC metadata includes an environment field — verify it reads production.
Large timestamp gaps Significant time gaps between sequential records may indicate removed or suppressed records. Compare against expected pipeline execution duration.
Signature key mismatch The signing key does not match the organization's registered key. May indicate unauthorized Evidence Pack generation or key compromise.
Missing cross-org evidence If the pipeline depends on upstream VPECs from other organizations, verify those VPECs are present and their chains are independently valid.

Tier 2 Audit — Activity Verification

Beyond Evidence Pack verification, auditors can request activity-level data for a deeper review.

Export Activity

Activity-level data can be exported via the API for deeper review:

# Export all runs for a workflow within a date range
curl -H "X-API-Key: $PRIMUST_API_KEY" \
  "https://primust-api.fly.dev/api/v1/runs?workflow_id=loan-underwriting-v3&from=2026-03-01&to=2026-03-12" \
  | jq . > activity-export.json

Verify Chain Integrity

Each VPEC contains a chain_intact boolean and chain_hash field. Verify chain integrity by checking each VPEC in the export:

# Verify each VPEC artifact from the export
for vpec in vpec_*.json; do
  primust-verify "$vpec" --production
done

Enterprise BYOK — Bring Your Own Key

Enterprise customers can use their own signing keys, anchored to an internal trust root. This means the organization's PKI — not Primust's — is the root of trust for Evidence Packs.

Trust Anchor Configuration

# In the organization's Primust config
{
  "trust_anchor_url": "https://pki.acme.corp/.well-known/primust-trust-anchor",
  "key_type": "Ed25519",
  "rotation_policy": "90d"
}

Verify with Custom Trust Root

# Auditor verifies against the org's trust root
primust verify evidence-pack-2026-03-12.json \
  --trust-root=https://pki.acme.corp/.well-known/primust-trust-anchor

When --trust-root is specified, primust-verify fetches the organization's public key from the specified URL and validates the signature chain against it, bypassing Primust's default key registry entirely.

Standard Audit Procedure

The following Python script implements a standard audit workflow: verify the Evidence Pack, extract the surface report, check for red flags, and generate a summary.

Pythonimport subprocess
import json
import sys

def audit_evidence_pack(pack_path, trust_root=None):
    """Standard audit procedure for a Primust Evidence Pack."""

    # Step 1: Verify the Evidence Pack
    cmd = ["primust-verify", pack_path, "--json"]
    if trust_root:
        cmd.extend(["--trust-root", trust_root])

    result = subprocess.run(cmd, capture_output=True, text=True)
    verification = json.loads(result.stdout)

    if verification["result"] != "PASS":
        print(f"FAIL: Verification failed — {verification['failures']}")
        sys.exit(1)

    # Step 2: Extract surface report
    pipeline = verification["pipeline"]
    # Surface coverage is embedded in the verification result
    surface = verification.get("coverage", {})

    # Step 3: Check for red flags
    flags = []
    if surface.get("policy_coverage_pct", 100) < 50:
        flags.append("LOW_COVERAGE: Less than 50% policy coverage")
    if verification["gaps"]["critical"] > 0:
        flags.append(f"CRITICAL_GAPS: {verification['gaps']['critical']} critical gaps")
    if any(w["expired"] for w in verification.get("waivers", [])):
        flags.append("EXPIRED_WAIVERS: One or more gap waivers have expired")
    if verification["environment"] == "sandbox":
        flags.append("SANDBOX_VPEC: Evidence Pack generated in sandbox mode")

    # Step 4: Output summary
    summary = {
        "pack": pack_path,
        "pipeline": pipeline,
        "verification": "PASS",
        "proof_coverage": surface,
        "gaps": verification["gaps"],
        "red_flags": flags,
        "recommendation": "CLEAR" if not flags else "REVIEW_REQUIRED"
    }

    print(json.dumps(summary, indent=2))
    return summary

if __name__ == "__main__":
    audit_evidence_pack(sys.argv[1])

Cross-Org Verification in Audit

When a pipeline depends on upstream data or computations from another organization, cross-org VPECs link the two chains of evidence. During audit, look for the following in the evidence field:

  • upstream_vpec_id — The VPEC identifier from the upstream organization
  • upstream_org_id — The upstream organization's identifier
  • upstream_signature — The upstream organization's signature over their VPEC
  • upstream_chain_hash — The final chain hash from the upstream VPEC, proving the upstream execution chain was intact at the time of linkage
  • verification_timestamp — When the downstream organization verified the upstream VPEC

To verify cross-org evidence independently:

# Verify a VPEC that references upstream cross-org VPECs
primust-verify vpec_cross-org.json --json

# The output includes upstream_vpec_ids and upstream_chain_hash.
# To verify the upstream chain independently, request the upstream VPEC
# from the counterparty and verify it with their trust root:
primust-verify upstream_vpec.json --trust-root partner-pubkey.pem

Audit Reports

Primust can generate signed PDF audit reports that bundle verification results, surface coverage, gap analysis, and framework alignment into a single document.

Generate a Signed Report

Signed audit reports are generated via the API or dashboard at app.primust.com.

Verify a Report

# Verify a signed PDF audit report
primust-verify audit-report-2026-03-12.pdf

# Or explicit report mode for JSON reports (deprecated format)
primust-verify audit-report.json --type report

Report Contents

  • Executive Summary — Pass/fail status, pipeline identification, date range, proof coverage percentage
  • Verification Results — All five checks with detailed output
  • Provable Surface Report — Full stage-by-stage coverage with proof levels
  • Gap Analysis — All gaps listed with taxonomy codes, severity, and waiver status
  • Framework Alignment — Mapping of Evidence Pack artifacts to applicable framework requirements
  • Red Flag Assessment — Automated red flag check results
  • Coverage Basis — P1/P2/P3 breakdown with per-stage classification
  • Cryptographic Signatures — Report signing key, Evidence Pack signing key, chain of trust
