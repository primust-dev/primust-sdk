Technical Reference

Complete specification for the Primust verification infrastructure — invariants, schemas, APIs, and circuits.

System Invariants

The following invariants hold for every Primust deployment. Violation of any invariant is treated as a system failure.

  1. Append-only chain. Once a check execution record is committed to a run's chain, it cannot be modified or removed. The chain is hash-linked; altering any record invalidates all subsequent hashes.
  2. No data transit. Primust never receives, stores, or transmits the actual data being processed. Only metadata, hashes, and proofs transit the system.
  3. Deterministic check execution. Given identical inputs and configuration, a check must produce identical outputs. Non-deterministic checks are rejected at registration.
  4. Proof level monotonicity. Within a single VPEC, a stage's proof level cannot be downgraded after initial commitment. Upgrades are permitted.
  5. Gap disclosure completeness. Every known gap in the provable surface must be disclosed in the VPEC metadata using the Gap Taxonomy. Undisclosed gaps constitute a system integrity failure.
  6. Key isolation. Signing keys are scoped to a single organization. Cross-org key usage is prohibited and detected by the verification system.
  7. Temporal ordering. Records within a run are strictly ordered by timestamp. Out-of-order records are rejected at commit time.
  8. Circuit immutability. Published ZK circuits are immutable. A new circuit version receives a new identifier; the old version remains available for verification of historical proofs.
  9. Offline verifiability. Any Evidence Pack can be verified without network access, given the signing public key and the ZK verification keys.
  10. Schema forward compatibility. Newer versions of primust-verify can verify Evidence Packs generated by older versions. Backward compatibility is maintained for a minimum of 24 months.
  11. Waiver expiration. Gap waivers have a mandatory expiration date. Expired waivers are surfaced as unresolved gaps in verification output.
  12. Single-writer runs. A run is owned by exactly one pipeline instance. Concurrent writes to the same run are rejected.
  13. Evidence Pack sealing. Once an Evidence Pack is generated, no additional records can be appended. The pack is cryptographically sealed at generation time.

Data Flow — What Transits Primust

Primust operates on metadata, never on the data itself. The following shows what Primust receives during a pipeline check execution:

JSON{
  "run_id": "run_8f3a2b1c",
  "check_id": "secrets_scanner",
  "stage": "data-ingest",
  "input_hash": "sha256:a1b2c3d4...",
  "output_hash": "sha256:e5f6g7h8...",
  "config_hash": "sha256:i9j0k1l2...",
  "result": "PASS",
  "proof_level": "P3",
  "zk_proof": "base64:...",
  "timestamp": "2026-03-12T14:22:18.003Z",
  "prev_record_hash": "sha256:m3n4o5p6...",
  "metadata": {
    "check_version": "1.2.0",
    "circuit_id": "secrets_scanner_v3",
    "execution_ms": 142
  }
}

What NEVER transits Primust:

  • Raw input data (documents, images, database rows, API payloads)
  • Model weights or parameters
  • Personally identifiable information (PII)
  • Protected health information (PHI)
  • Trade secrets or proprietary algorithms
  • Credentials, tokens, or API keys
  • Plaintext outputs or intermediate computation results

BYOK Signing Spec

Database Schema

SQLCREATE TABLE signing_keys (
  key_id        TEXT PRIMARY KEY,
  org_id        TEXT NOT NULL REFERENCES organizations(org_id),
  public_key    BYTEA NOT NULL,
  key_type      TEXT NOT NULL DEFAULT 'Ed25519',
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at    TIMESTAMPTZ,
  trust_anchor  TEXT,
  fingerprint   TEXT NOT NULL UNIQUE,
  metadata      JSONB DEFAULT '{}'
);

Registration Flow

  1. Generate key pair. Organization generates an Ed25519 key pair locally. The private key never leaves the organization's infrastructure.
  2. Register public key. POST /api/v1/keys with the public key, optional trust anchor URL, and key metadata.
  3. Challenge-response. Primust issues a random nonce. The organization signs the nonce with the private key and returns the signature. Primust verifies it against the registered public key.
  4. Activation. Key is marked active and available for Evidence Pack signing.
  5. Rotation. When rotation policy triggers, repeat steps 1-4 with a new key pair. The old key remains valid for verification of historical packs but cannot sign new ones.

Verification Spec

Four Verification Paths

PathUse CaseRequirements
Online (default)Standard verification with key registry lookupNetwork access to Primust key registry
OfflineAir-gapped or restricted environmentsLocal copy of org public key + ZK verification keys
BYOK Trust RootEnterprise customers with their own PKINetwork access to org's trust anchor URL
Cross-OrgVerifying upstream VPECs from other organizationsNetwork access to Primust registry for upstream key lookup

Eight Verification Steps

  1. Parse Evidence Pack. Validate JSON structure against VPEC schema.
  2. Resolve signing key. Fetch org public key from registry, trust root, or local file.
  3. Verify signature. Validate Ed25519 signature over the sealed pack contents.
  4. Walk the chain. Starting from the first record, verify each prev_record_hash link. Confirm chain_root_hash matches.
  5. Verify ZK proofs. For each P3+ record, load the circuit verification key and verify the proof.
  6. Check timestamps. Confirm monotonic ordering and that all timestamps fall within the declared execution window.
  7. Validate gaps. Confirm all gaps are from the Gap Taxonomy, check waiver expiration, count critical gaps.
  8. Produce result. Emit PASS/FAIL with detailed findings for each step.

Exit Codes

CodeMeaning
0PASS — all checks passed
1FAIL — one or more verification checks failed
2ERROR — could not complete verification (missing key, corrupt file, network error)
3WARN — passed with warnings (expired waivers, low proof coverage)

Offline Operation

# Download verification keys for offline use
primust keys export --org=org_x9y8z7 --output=org-keys.json

# Download ZK verification keys
primust circuits export --output=circuit-keys.json

# Verify offline
primust verify evidence-pack.json \
  --offline \
  --keys=org-keys.json \
  --circuits=circuit-keys.json

Audit Report Verification

# Verify a signed PDF audit report
primust verify-report audit-report-2026-03-12.pdf

# Output:
# ✓ PDF signature valid
# ✓ Embedded evidence hash matches source pack
# ✓ Report signing key is authorized
# RESULT: PASS

Gap Taxonomy

The Gap Taxonomy defines every recognized category of coverage gap. Gaps are classified into three groups: Core (pipeline-level), Unstructured (data-level), and Cross-Org (multi-party).

Core Gaps (22)

IDNameSeverityDescription
G-01Undeclared StageCriticalA pipeline stage exists but is not included in the VPEC definition
G-02Missing CheckHighA declared stage has no check assigned to it
G-03Stale ManifestMediumThe policy bundle manifest has not been updated in over 90 days
G-04Proof DowngradeHighA stage was previously at a higher proof level and has been downgraded
G-05Missing ZK CircuitCriticalA stage declares P3 but no ZK circuit is assigned or available
G-06Unsigned RecordsCriticalOne or more records in the chain lack a valid signature
G-07Clock SkewMediumTimestamps show skew exceeding the configured tolerance
G-08Config DriftMediumCheck configuration hash differs from the manifest-declared hash
G-09Dependency MismatchHighRuntime dependency hashes do not match declared values
G-10Partial CoverageMediumA stage check covers only a subset of the stage's inputs
G-11Stale CircuitLowCircuit version is more than two major versions behind current
G-12Missing Error HandlerMediumPipeline error paths are not instrumented
G-13Orphan RecordsHighRecords exist that do not belong to any declared stage
G-14Duplicate SequenceCriticalTwo records share the same sequence number within a run
G-15Key ExpiringLowSigning key will expire within 30 days
G-16Waiver PendingMediumA gap waiver request has been submitted but not yet approved
G-17Logging GapLowObservability or logging pipeline stages are not instrumented
G-18Retry UncoveredMediumRetry/fallback paths in the pipeline are not instrumented
G-19Batch BoundaryMediumBatch processing boundaries are not individually tracked
G-20Cache Hit PathMediumCached results bypass the normal check execution path
G-21Feature Flag PathMediumFeature-flagged code paths may execute different logic without separate checks
G-22SDK Version MismatchLowSDK version does not match the recommended version for the engine
