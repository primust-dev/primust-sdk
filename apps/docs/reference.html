Technical Reference

Complete specification for the Primust verification infrastructure — invariants, schemas, APIs, and gap taxonomy.

System Invariants

The following invariants hold for every Primust deployment. Violation of any invariant is treated as a system failure.

  1. No data transit. Raw content (inputs, outputs, PII, model weights) NEVER leaves the customer environment. Only commitment hashes (poseidon2 or sha256) transit to api.primust.com.
  2. Append-only chain. Once a check execution record is committed to a run's chain, it cannot be modified or removed. The chain is hash-linked; altering any record invalidates all subsequent chain hashes.
  3. Deterministic commitments. Given identical input bytes, commit() always produces the same hash. Different inputs always produce different hashes.
  4. Weakest-link proof level. A VPEC's overall proof level is the lowest proof level of any record in the run. One attestation record forces the entire VPEC to attestation.
  5. Gap disclosure completeness. Every known gap in the provable surface must be disclosed in the VPEC using the canonical gap taxonomy. Undisclosed gaps constitute a system integrity failure.
  6. Single-writer runs. A run is owned by exactly one pipeline instance. Concurrent writes to the same run are rejected.
  7. Temporal ordering. Records within a run are strictly ordered by sequence number. check_open_tst is captured at openCheck()/open_check() time; check_close_tst at record() time.
  8. Offline verifiability. Any VPEC can be verified without network access using primust-verify --skip-network, given the signing public key.
  9. Schema forward compatibility. Newer versions of primust-verify can verify VPECs generated by older schema versions. Current schema version: 4.0.0.
  10. Key isolation. Signing keys are scoped to a single organization. Cross-org key usage is prohibited and detected by the verification system.
  11. Evidence Pack sealing. Once a VPEC is issued (run.close()), no additional records can be appended. The credential is cryptographically sealed at issuance time.

Proof Levels

Every check execution record declares a proof_level_achieved. The five canonical levels, from strongest to weakest:

LevelEnum ValueWhat It ProvesWhen Used
MathematicalmathematicalZK circuit proves check execution without revealing inputsZK proof generated and verified
Verifiable Inferenceverifiable_inferenceModel inference with verifiable execution traceZKML proof of model inference
ExecutionexecutionCheck code ran in an instrumented environment with commitment hashingDefault for SDK record() calls
WitnessedwitnessedA human reviewer signed the check result with a reviewer credentialopenReview() + reviewer_signature
AttestationattestationOrganization self-declares the check passed — no cryptographic proofManual/imported records without signatures

The VPEC's overall proof_level is the weakest level seen across all records (weakest-link rule).

Data Flow — What Transits Primust

The SDK commits input locally via commit() and sends only the resulting hash to the API. Here is what a check execution record looks like on the wire:

JSON{
  "manifest_id": "pii_scan_manifest",
  "commitment_hash": "sha256:a1b2c3d4e5f6...",
  "commitment_algorithm": "sha256",
  "commitment_type": "input_commitment",
  "output_commitment": "sha256:f6e5d4c3b2a1...",
  "check_result": "pass",
  "proof_level_achieved": "execution",
  "check_open_tst": "2026-03-12T14:22:18.003Z",
  "check_close_tst": "2026-03-12T14:22:18.145Z",
  "idempotency_key": "idem_a1b2c3d4e5f67890"
}

What NEVER transits Primust:

  • Raw input data (documents, images, database rows, API payloads)
  • Raw output data (model responses, computation results)
  • Model weights or parameters
  • Personally identifiable information (PII)
  • Protected health information (PHI)
  • Credentials, tokens, or API keys
  • Display content, rationale text, or skip rationale text (only hashes transit)

BYOK Registration Flow

  1. Generate key pair. Organization generates an Ed25519 key pair locally. Private key never leaves org infrastructure.
  2. Register public key. POST /api/v1/org/signing-keys with the public key.
  3. Challenge-response. Primust issues a nonce; org signs it; POST .../verify confirms ownership.
  4. Activation. Key marked active, available for Evidence Pack and audit report signing.
  5. Rotation. Register new key, old key remains valid for verification of historical packs.

Gap Taxonomy

The gap taxonomy defines every recognized category of coverage gap. Gaps are classified into three groups: Core (pipeline-level), Unstructured (data-level), and Cross-Org (multi-party). Each gap has a machine-readable gap_type string used in the VPEC gaps array.

Verification

primust-verify CLI

primust-verify is a standalone binary (Apache-2.0, free forever, no account required) that verifies VPECs offline.

# Verify a VPEC
primust-verify vpec.json

# Production mode (stricter checks)
primust-verify --production vpec.json

# Fully offline (no network key lookups)
primust-verify --skip-network --trust-root org-key.pub vpec.json

# JSON output for CI/CD pipelines
primust-verify --json vpec.json

Verification Steps

  1. Parse VPEC. Validate JSON structure against schema v4.0.0.
  2. Resolve signing key. Fetch org public key from registry or use --trust-root local key.
  3. Verify signature. Validate Ed25519 signature over the sealed VPEC contents.
  4. Walk the chain. Replay chain hashes from the first record forward. Confirm commitment_root matches.
  5. Check commitments. Verify all commitment_hash values use the declared commitment_algorithm.
  6. Validate timestamps. Confirm monotonic ordering of check_open_tst / check_close_tst pairs.
  7. Validate gaps. Confirm all gaps use canonical gap_type values from the taxonomy.
  8. Produce result. Emit PASS/FAIL with detailed findings for each step.

Exit Codes

CodeMeaning
0PASS — all verification checks passed
1FAIL — one or more verification checks failed
2ERROR — could not complete verification (missing key, corrupt file, network error)

SIEM Linkage

The SDK provides a logger callback that emits structured events for your existing logging infrastructure (Splunk, Datadog, CloudWatch, etc.). This creates an auditable linkage between your application logs and the Primust VPEC.

Python

Pythonimport logging
logger = logging.getLogger("myapp")

def primust_logger(event):
    logger.info("primust_record", extra={
        "primust_commitment_hash": event.primust_commitment_hash,
        "primust_check_result": event.primust_check_result,
        "primust_run_id": event.primust_run_id,
    })

pipeline.set_logger(primust_logger)

TypeScript

TypeScriptpipeline.setLogger((event) => {
  console.log("primust_record", {
    primust_commitment_hash: event.primust_commitment_hash,
    primust_check_result: event.primust_check_result,
    primust_run_id: event.primust_run_id,
  });
});

Auditor Verification via SIEM

  1. Search SIEM: WHERE primust_commitment_hash = <value>
  2. Run: primust-verify hash <plaintext_input> to confirm hash matches
  3. VPEC proves chain integrity, timestamp independence, and proof level
