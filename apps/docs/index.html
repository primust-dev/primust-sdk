Primust Developer Documentation

Prove what your system did. Mathematically.

What is Primust

You already have checks—access control, schema validation, model evaluation, bias audits, policy enforcement. Primust does not replace them. Primust makes them provable.

The flow is simple:

  1. Input — your system receives a request, data, or event.
  2. Checks — your existing governance logic executes: OPA policies, Cedar rules, model evaluations, schema validators, anything.
  3. Output — Primust captures the execution trace, binds it cryptographically, and produces a credential.
  4. Verify — anyone with the credential can verify it independently. No phone calls. No network access. No trust required.

The credential is called a VPEC—a Verifiable Process Execution Credential. It is portable JSON. It is verifiable offline. It contains commitment hashes, never raw content. It is the mathematical receipt for what your system actually did.

What is a VPEC

Verifiable Process Execution Credential.

  • Verifiable — any party can independently confirm the credential is valid, without contacting the issuer.
  • Process — it binds to a specific execution: the checks that ran, the inputs they consumed, the outputs they produced.
  • Execution — it captures what actually happened at runtime, not what was supposed to happen.
  • Credential — it is a portable, self-contained JSON object that travels with the artifact it describes.

A VPEC is cryptographically signed JSON. It contains commitment hashes—SHA-256 digests of inputs, outputs, and check results—never the raw content itself. This makes VPECs content-blind: a verifier can confirm that a bias audit passed without ever seeing the model weights or training data.

VPECs are portable. Send them across organizations, store them in object storage, embed them in API responses. They are offline-verifiable—verification requires only the credential and a trust root, no network calls.

Key Property

A VPEC proves what happened, not what should have happened. It is a receipt, not a policy.

Proof Levels

Not all evidence is equal. Primust assigns every check a proof level based on the cryptographic strength of its binding. Five levels, from strongest to weakest:

Level Mechanism Trust Assumption
Mathematical Zero-knowledge proof (ZK circuit) None — no trust required
Verifiable Inference ONNX-to-circuit compilation Trust the model weights
Execution Model-hash-binding (deterministic replay) Trust the model
Witnessed Dual timestamp + Ed25519 signature Trust the reviewer
Attestation Invocation-binding (call graph capture) Trust the process owner
Weakest-Link Rule

A VPEC's effective proof level is the lowest proof level among all checks it contains. If nine checks are Mathematical and one is Attestation, the VPEC is Attestation-level. This is by design—a chain is only as strong as its weakest link.

Provable Surface

The provable surface is the percentage of your governance that is cryptographically provable. It answers a simple question: of all the governance checks in this process, how many produce verifiable evidence?

A sample breakdown for a typical AI agent pipeline:

Check Proof Level Provable
Input schema validation Mathematical Yes
OPA policy evaluation Execution Yes
Bias audit (ONNX model) Verifiable Inference Yes
Human review sign-off Witnessed Yes
LLM output toxicity scan Attestation Yes
Manual spreadsheet check — No
Verbal approval — No

Provable surface: 71% (5 of 7 checks produce verifiable evidence). The two non-provable checks represent governance gaps—Primust flags these explicitly so you can decide whether to instrument them or accept the gap.

Policy Bundles

Policy bundles are pre-built collections of checks mapped to specific compliance frameworks or use cases. Each bundle defines which checks are required, their minimum proof levels, and the expected provable surface.

Bundle ID Description Checks
ai_agent_general_v1 General-purpose AI agent governance Input validation, output filtering, tool-call authorization, execution trace
eu_ai_act_art12_v1 EU AI Act Article 12 transparency & logging Logging completeness, explanation commitment, bias audit, risk classification
hipaa_safeguards_v1 HIPAA technical safeguards for AI systems PHI detection, access control, audit log integrity, encryption verification
soc2_cc_v1 SOC 2 common criteria controls Access control, change management, risk assessment, monitoring
coding_agent_v1 Code-generating agent governance SAST binding, dependency audit, license check, secrets scan
supply_chain_governance_v1 Software supply chain integrity SBOM verification, provenance attestation, signature validation
financial_data_governance_v1 Financial data processing controls PII detection, regulatory hold, audit trail, reconciliation check

Bundles are composable. Apply multiple bundles to a single process to cover overlapping regulatory requirements. Use primust bundle list to see all available bundles and primust bundle inspect <id> to view the full check manifest.
