Quickstart

From zero to your first verifiable credential in under five minutes.

Install

Python (pip)

pip install primust

Node.js (npm)

npm install @primust/sdk

Java (Maven)

<dependency>
  <groupId>com.primust</groupId>
  <artifactId>primust-sdk</artifactId>
  <version>0.9.0</version>
</dependency>

Get a Sandbox Key

Sign up at primust.com and create a sandbox project. Your API key will look like pk_sb_xxx.

export PRIMUST_API_KEY=pk_sb_xxx
Sandbox vs Production

Sandbox keys (pk_sb_) are free and rate-limited. Production keys (pk_prod_) require a paid plan and enable archival, SLA guarantees, and cross-org verification.

Your First VPEC

Wrap your existing agent or pipeline with Primust to produce a verifiable credential. Choose your framework:

LangGraph

Pythonfrom primust import govern
from langgraph.graph import StateGraph

graph = StateGraph(...)
# ... define your nodes and edges ...

# Wrap compilation with Primust governance
app = govern(
    graph.compile(),
    bundle="ai_agent_general_v1",
    api_key="pk_sb_xxx"
)

result = app.invoke({"input": "Summarize this contract"})
vpec = result.vpec  # Verifiable Process Execution Credential
print(vpec.proof_level)   # e.g. "execution"
print(vpec.surface)       # e.g. 0.85

Discover Your Existing Checks

Primust can scan your codebase and infrastructure to find governance checks you already have:

Shellprimust discover --path ./src --output manifest.json

The discover command identifies OPA policies, Cedar schemas, validation decorators, model evaluation calls, and other governance-related patterns. It produces a manifest you can review and edit before registering.

Register a Custom Check

Using the Decorator

Pythonfrom primust import check

@check(
    name="pii_redaction",
    proof_level="execution",
    description="Verify PII is redacted before storage"
)
def redact_pii(text: str) -> str:
    # Your existing redaction logic
    redacted = your_redaction_fn(text)
    return redacted

Manual Manifest Registration

Verify a VPEC

Shell# Verify a credential
primust verify vpec_abc123.json

# Output:
# VPEC vpec_abc123
#   Status:       VALID
#   Proof Level:  execution
#   Surface:      85%
#   Checks:       12 passed, 0 failed, 2 gaps
#   Issued:       2026-03-15T08:30:00Z
#   Issuer:       org:acme-corp
Shell# Verify with an explicit trust root (offline verification)
primust verify vpec_abc123.json \
  --trust-root ./acme-corp-root.pub
Offline Verification

With --trust-root, verification requires zero network access. The credential, the trust root, and the primust-verify binary are all you need.

Assemble Evidence Pack

An evidence pack bundles multiple VPECs with metadata for audit submission:

Shell# Assemble all VPECs from the last 30 days
primust evidence assemble \
  --from 2026-02-15 \
  --to 2026-03-15 \
  --bundle eu_ai_act_art12_v1 \
  --output evidence-pack-q1.json

# List contents
primust evidence inspect evidence-pack-q1.json

Export Audit Report

Shell# Generate a human-readable PDF audit report
primust audit export \
  --evidence evidence-pack-q1.json \
  --format pdf \
  --output audit-report-q1.pdf

# Generate machine-readable JSON
primust audit export \
  --evidence evidence-pack-q1.json \
  --format json \
  --output audit-report-q1.json

Handle Gaps Honestly

Not every governance check can be made provable today. Primust treats gaps as first-class objects—they are declared, tracked, and surfaced in every VPEC and audit report.

Pythonfrom primust import gap

# Declare a known governance gap
gap(
    id="manual_review_verbal",
    description="Final approval is verbal and not captured digitally",
    category="process",
    severity="medium",
    mitigation="Migrating to signed digital approval by Q3 2026"
)

# The gap appears in every VPEC that touches this process.
# Auditors see it. Downstream verifiers see it.
# Honesty is the policy.

AIUC-1 Compliance Fields

When using the eu_ai_act_art12_v1 bundle, the following fields are required on every governed invocation:

Field Type Description
retention_policy string Data retention duration and policy reference (e.g. "90d:GDPR-Art17")
risk_classification enum EU AI Act risk tier: minimal, limited, high, unacceptable
actor_id string Pseudonymized identifier of the human or system that initiated the process
explanation_commitment hash SHA-256 of the explanation text provided to the end user
bias_audit object Reference to the most recent bias audit result, including method and date 
Pythonresult = governed_agent.run(
    "Classify this applicant",
    primust_fields={
        "retention_policy": "90d:GDPR-Art17",
        "risk_classification": "high",
        "actor_id": "usr:a1b2c3",
        "explanation_commitment": "sha256:e3b0c44298fc...",
        "bias_audit": {
            "method": "demographic_parity",
            "date": "2026-03-01",
            "result": "pass"
        }
    }
)

Receiving VPECs — The Downstream Pattern

When your system receives a VPEC from another organization, you verify it and then record the verification itself as a Mathematical-level check in your own VPEC:

Pythonfrom primust import verify, check

# Step 1: Verify the incoming VPEC
upstream_vpec = load_vpec("vendor-vpec.json")
result = verify(upstream_vpec, trust_root="./vendor-root.pub")

assert result.valid, f"Upstream VPEC invalid: {result.reason}"

# Step 2: Record the verification as a check in your pipeline
@check(
    name="upstream_vpec_verification",
    proof_level="mathematical",
    description="Cryptographic verification of vendor VPEC"
)
def verify_upstream():
    return {
        "upstream_vpec_id": upstream_vpec.id,
        "issuer": upstream_vpec.issuer,
        "proof_level": upstream_vpec.proof_level,
        "verified_at": result.timestamp,
        "trust_root": "vendor-root.pub"
    }

This creates a chain of trust across organizational boundaries. Your VPEC now contains a Mathematical-level check proving that you verified the upstream credential. The downstream party can verify your VPEC without needing to contact the original vendor.

Production Checklist

  • Swap sandbox key (pk_sb_) for production key (pk_prod_)
  • Enable VPEC archival with primust config set archival.enabled true
  • Set proof level floor: primust config set proof_level_floor execution
  • Configure trust root rotation schedule
  • Register all custom checks in the manifest
  • Run primust discover and resolve all flagged gaps
  • Enable cross-org verification if receiving external VPECs
  • Set retention policy on all governed endpoints
  • Integrate primust audit export into your CI/CD pipeline
  • Test offline verification with primust verify --trust-root
  • Review provable surface target and set alerts for regression
  • Configure alerting for VPEC verification failures

Key Concepts

Term Definition
GEP Governance Evidence Pack — a bundle of VPECs assembled for audit submission, covering a specific time range and compliance framework.
VPEC Verifiable Process Execution Credential — a cryptographically signed JSON object proving what checks ran and what they produced.
Manifest A JSON declaration of all checks in a project, including their types, proof levels, and entry points. Generated by primust discover or authored manually.
Provable Surface The percentage of governance checks that produce cryptographically verifiable evidence. Higher is better; 100% means every check is provable.
Proof Level Floor The minimum acceptable proof level for a deployment. If any check falls below the floor, the VPEC is flagged. Configured per environment.
Archetype A reusable governance pattern for a common use case (e.g. "LLM agent with tool access", "batch data pipeline"). Archetypes map to one or more policy bundles.
