Quickstart

From zero to your first verifiable credential in under five minutes.

Install

Python (pip)

pip install primust

Node.js (npm)

npm install @primust/sdk

Framework Adapters

# External adapter packages (install alongside primust)
pip install primust-langgraph        # LangGraph
pip install primust-openai-agents    # OpenAI Agents SDK
pip install primust-google-adk       # Google ADK
pip install primust-otel             # OpenTelemetry

# Built-in adapters (included with primust, no extra install)
# primust.adapters.crewai
# primust.adapters.pydantic_ai

Verifier

# Standalone verifier — no SDK dependency required
pip install primust-verify
# or
npm install @primust/verifier

Get an API Key

Sign up at primust.com and create a project. Your test API key will look like pk_test_xxx.

export PRIMUST_API_KEY=pk_test_xxx
Test vs Live Keys

Test keys (pk_test_) are free and rate-limited. Live keys (pk_live_) require a paid plan and enable archival, SLA guarantees, and cross-org verification.

Your First VPEC

The core SDK exposes a Pipeline that you open checks against, record results, and close to produce a VPEC. All framework adapters wrap this same Pipeline.

Core SDK — Python

Pythonfrom primust import Pipeline

p = Pipeline(api_key="pk_test_xxx", workflow_id="wf_onboard")

# Open a check, record a result, close the pipeline
session = p.open_check("pii_check", "manifest_001")
result = p.record(session, input=my_data, check_result="pass")

vpec = p.close()   # returns dict — your Verifiable Process Execution Credential
print(vpec)

Core SDK — JavaScript

JavaScriptimport { Pipeline } from '@primust/sdk';

const p = new Pipeline({ apiKey: 'pk_test_xxx', workflowId: 'wf_onboard' });

const session = await p.openCheck('pii_check', 'manifest_001');
const result = await p.record(session, input, 'pass', { output: resultData });

const vpec = await p.close();
console.log(vpec);

Pipeline API

Every adapter is a thin wrapper around the same Pipeline object. Here is the full surface:

Method Signature
Pipeline() Pipeline(api_key, workflow_id, *, policy=None, process_context_hash=None, base_url="https://api.primust.com")
open_check p.open_check(check_name, manifest_id) → CheckSession
open_review p.open_review(check_name, manifest_id, reviewer_key_id, min_duration_seconds=1800) → ReviewSession
record p.record(session, input, check_result, *, output=None, reviewer_signature=None, display_content=None, rationale=None, skip_rationale=None) → RecordResult
close p.close(*, partial=False, request_zk=False) → dict (the VPEC)
JS SDK

The JavaScript SDK (@primust/sdk) exposes the same API in camelCase: openCheck, openReview, record, close. All methods return Promises.

Verify a VPEC

The verifier is a separate package — install it with pip install primust-verify or npm install @primust/verifier. The binary is called primust-verify.

Shell# Verify a credential
primust-verify vpec_abc123.json

# Output:
#   ✓ VPEC vpec_abc123 — VALID
#     Proof level:   execution (weakest-link)
#     Distribution:  mathematical: 3  execution: 9
#     Workflow:      wf_onboard
#     Org:           org_acme
#     Signed:        2026-03-15T08:30:00Z
#     Coverage:      85% policy
#     Gaps:          2 (1 Medium, 1 Low)
#     Test mode:     false
Shell# Production mode (stricter validation)
primust-verify vpec_abc123.json --production

# Offline verification with an explicit trust root
primust-verify vpec_abc123.json --skip-network --trust-root ./acme-corp-root.pub

# Machine-readable JSON output
primust-verify vpec_abc123.json --json
Exit Codes

0 = valid, 1 = invalid, 2 = system error. Use these in CI/CD pipelines to gate deployments on VPEC validity.

Offline Verification

With --skip-network --trust-root, verification requires zero network access. The credential, the trust root, and the primust-verify binary are all you need.

Receiving VPECs — The Downstream Pattern

When your system receives a VPEC from another organization, verify it using the standalone verifier and then record that verification as a check in your own pipeline:

Shell# Step 1: Verify the upstream VPEC
primust-verify vendor-vpec.json --trust-root ./vendor-root.pub --json > verification.json
Pythonimport json
from primust import Pipeline

# Step 2: Record the verification as a check in your pipeline
p = Pipeline(api_key="pk_test_xxx", workflow_id="wf_ingest")

with open("verification.json") as f:
    upstream = json.load(f)

session = p.open_check("upstream_vpec_verification", "manifest_vendor_v1")
p.record(
    session,
    input=upstream,
    check_result="pass" if upstream["valid"] else "fail",
    output={"upstream_vpec_id": upstream["vpec_id"], "proof_level": upstream["proof_level"]}
)

# ... run the rest of your pipeline checks ...

vpec = p.close()

This creates a chain of trust across organizational boundaries. Your VPEC now contains a check proving you verified the upstream credential. The downstream party can verify your VPEC without needing to contact the original vendor.

Production Checklist

  • Swap test key (pk_test_) for live key (pk_live_)
  • Set base_url in Pipeline constructor if using a dedicated endpoint
  • Use open_review for any human-in-the-loop checks (enforces minimum review duration)
  • Pass request_zk=True to p.close() for checks that need zero-knowledge proofs
  • Configure trust root rotation schedule
  • Define all checks in manifest files and reference them by manifest_id
  • Set up cross-org verification if receiving external VPECs
  • Run primust-verify --production in CI/CD to gate on VPEC validity
  • Test offline verification with primust-verify --skip-network --trust-root
  • Configure alerting for VPEC verification failures (exit code 1)
  • Store VPECs in durable, append-only storage for audit trail
  • Set policy parameter in Pipeline constructor to bind runs to a specific policy version

Key Concepts

Term Definition
VPEC Verifiable Process Execution Credential — a cryptographically signed JSON object proving what checks ran and what they produced.
Pipeline The core SDK object. You open checks, record results, and close to produce a VPEC. All framework adapters wrap a Pipeline.
CheckSession A handle returned by open_check, representing an in-progress check within the pipeline. Passed to record to bind a result.
ReviewSession Like CheckSession but for human reviews. Returned by open_review, enforces minimum review duration and requires a reviewer signature.
Manifest A versioned identifier (manifest_id) for a check definition. Each check session is opened against a specific manifest.
Proof Level The strength of evidence for a check. In ascending order: attestation, witnessed, execution, verifiable_inference, mathematical.
Adapter A thin wrapper that instruments a specific framework (LangGraph, OpenAI Agents, CrewAI, etc.) to automatically open checks, record results, and produce VPECs via a Pipeline.
